# Claude Code (ralph-loop) vs 传统工具：参数校准任务分层分析

> 2026-03-26 | Tommy-2 深度分析

## Executive Summary

**核心结论：** 当前校准任务（3 参数、210 组合、43 场景）处于 Level 1 和 Level 2 的交界地带。传统工具（网格搜索/Optuna）在纯参数搜索上效率碾压 Claude Code，但 Claude Code 的独特价值在于它能在搜索过程中**发现问题并跨越到结构诊断**——这是传统工具完全无法做到的。最优策略不是二选一，而是分层组合：用传统工具做搜索引擎，用 Claude Code 做诊断大脑。

---

## 一、任务分层分析

### Level 1 — 参数调优（固定公式，搜索最优值）

**任务特征：** 搜索空间已定义（_K, _GAMMA, _OPP_NEG），目标函数已定义（MAD + sent_match），公式结构不变。

**最适合的工具：传统优化工具（网格搜索 / Optuna TPE）**

理由：

| 维度 | 传统工具 | Claude Code (ralph-loop) |
|------|----------|--------------------------|
| 搜索效率 | 210 组合网格搜索 < 1 秒完成 | 每轮 1 次 LLM 调用 x 多轮迭代，10+ 分钟 |
| 精确性 | 浮点计算精确到机器精度 | LLM 数学计算偶有误差（见下文分析） |
| 可复现性 | 100% 确定性 | 受 prompt 理解、上下文窗口限制 |
| 搜索策略 | Optuna TPE 自适应采样，数学最优 | 启发式"看哪个方向好就往那走" |
| 扩展性 | 从 3 参数扩到 10 参数，Optuna 仍可处理 | 参数越多，LLM 心智负担越重 |

**关键事实：** 项目中 `threshold_optimizer.py` 已经有成熟的 Optuna TPE 搜索框架（支持 Beam Search warm start、checkpoint 续跑、shrinkage score），这不是需要从零搭建的基础设施。当前校准任务的目标函数（MAD + sent_match）比 threshold_optimizer 的 shrinkage score 还简单。

**Claude Code 在 Level 1 的价值：接近于零。** 让 LLM 手动做 210 次公式计算，本质上是用 GPU 算力来模拟 CPU 算力，效率差 3-4 个数量级。ralph-loop 的"每轮自主决策"能力在纯数值搜索中无用武之地——没有什么需要"决策"的，遍历就是了。

### Level 2 — 结构诊断（分析公式是否合理，发现系统性缺陷）

**任务特征：** 不是"找最优参数"，而是"判断公式结构能否覆盖所有场景"。

**最适合的工具：Claude Code（非 ralph-loop，而是单次深度分析）**

理由：

结构诊断需要的能力恰好是 LLM 的强项，传统工具的盲区：

1. **语义理解残差分析。** 传统工具只能告诉你"场景 27 的残差最大"，但无法告诉你"场景 27 的 5P(low)+2N(medium)+3U(low) 组合暴露了 scarcity 保护和 opposition penalty 之间的交互缺陷"。Claude Code 能读懂场景语义，将数值残差映射到公式的具体子项。

2. **跨场景模式识别。** "所有包含 extreme 单条 + 多条 low 反方向的场景都系统性偏高" ——这种模式需要同时理解场景结构和公式逻辑，不是统计回归能发现的。

3. **因果推理。** 残差大不代表公式有 bug——可能是 ground truth 标注有问题（calibration_path_analysis.md 中场景 27 的分析就是例证）。分辨"公式缺陷"和"标注噪声"需要领域知识，传统工具无法提供。

**但 ralph-loop 迭代模式不是必需的。** 结构诊断本质上是一次性的深度分析任务，不是迭代收敛问题。一次高质量的分析 prompt 比 10 轮浅层迭代更有效。

### Level 3 — 公式重构（修改数学结构）

**任务特征：** 增删公式子项、改变函数形式（如将 opposition penalty 从线性改为非线性）。

**最适合的工具：人类决策 + Claude Code 辅助实现**

理由：

1. **公式结构承载业务意图。** `_summarize` 的三分支设计（positive/negative/冲突 neutral vs 共识 neutral）不是数学优化的结果，而是对"新闻情感聚合应该如何工作"这个业务问题的回答。改结构 = 改业务假设，这需要人类判断。

2. **Claude Code 的辅助价值。** 一旦人类决定"opposition penalty 应该是非线性的"，Claude Code 可以：(a) 提出具体的数学形式候选（如幂函数、sigmoid），(b) 分析每种形式的渐近行为，(c) 实现并验证。但它不应该自主决定是否要改。

3. **传统工具完全不适用。** Optuna 能搜索参数但不能搜索函数形式。遗传编程（symbolic regression）理论上可以搜索函数结构，但需要大量训练数据且可解释性差，不适合这个场景。

## 二、Claude Code + ralph-loop 的独特价值评估

### ralph-loop 的核心能力

ralph-loop 是一个迭代框架，每轮 AI 自主决定做什么。它的三个独特能力：

1. **读代码-写代码-运行代码** 的完整循环
2. **分析中间结果并调整策略** 的自适应能力
3. **跨越"分析"和"实现"的边界** 的灵活性

### 在三个层次中的价值

| 能力 | Level 1 (参数调优) | Level 2 (结构诊断) | Level 3 (公式重构) |
|------|-------------------|-------------------|-------------------|
| 读/写/运行代码 | 低：纯计算任务，Python 脚本 1 秒完成 | 中：需要读公式代码理解逻辑 | 高：需要修改代码并验证 |
| 分析+调整策略 | 低：搜索空间小，无需策略调整 | 高：残差模式分析需要动态深挖 | 高：需要评估修改效果并迭代 |
| 跨越分析/实现 | 低：纯搜索无需跨越 | 中：诊断后可直接指出修复方向 | **极高：这是唯一能力** |

**核心洞察：** ralph-loop 的真正独特价值出现在 Level 2→3 的跨越中——当诊断发现结构性问题时，它可以直接从"发现问题"过渡到"修改代码并验证修改效果"。这是传统工具链（Optuna 搜索 → 人工分析残差 → 手动修改代码 → 重新搜索）中最耗时的环节。

但要注意：**当前的 calibrate_prompt.md 把 ralph-loop 钉死在 Level 1 上了。** 它明确指定了搜索参数、搜索范围、终止条件，不允许 AI 跨越到结构诊断。这恰恰浪费了 ralph-loop 的独特能力，同时把它用在了效率最差的层次上。

## 三、开放式校准的交付物定义

### 四个选项分析

| 选项 | 内容 | 风险 | 适合阶段 |
|------|------|------|---------|
| A | 最优参数值 | 低风险，但忽略结构问题 | 成熟公式的微调 |
| B | 参数值 + 诊断报告 | 低风险，信息量最大 | **当前阶段最优** |
| C | 参数值 + 诊断 + 改进建议（代码级） | 中风险，建议可能不够好 | 有明确结构问题时 |
| D | 直接修改后的代码 + 验证 | 高风险，AI 自主改公式 | 需要人类审核把关 |

### 推荐：Option B（当前阶段最优）

**理由：**

1. **项目阶段因素。** `_summarize` 公式刚经历了 confidence→impact 的迁移（见 calibration_path_analysis.md 中的背景），正处于"公式结构可能需要调整"的窗口期。此时纯参数调优（Option A）可能掩盖结构性问题。

2. **calibration_path_analysis.md 的发现支持这一判断。** 该报告已经识别出标注与公式之间的循环论证问题，说明当前的校准流程本身就在 Level 1 和 Level 2 之间纠结。需要诊断报告来明确"参数调优够了"还是"需要结构修改"。

3. **Option C/D 的前提条件未满足。** 在没有真实数据验证（Step 5 尚未执行）的情况下，让 AI 提出代码级改进建议缺乏足够的信号支撑。诊断可以基于合成场景做，但修改建议需要真实数据分布的校验。

## 四、风险-收益矩阵

|  | 低风险 | 高风险 |
|---|---|---|
| **高收益** | 网格搜索 3 参数; AI 诊断报告（不改代码） | AI 修改单个子项（如 opp_penalty 计算方式） |
| **低收益** | 扩大搜索到所有参数 | AI 自主决定搜索哪些参数; AI 重写整个 _summarize |

### 逐项分析

**网格搜索 3 参数 — 低风险/高收益**

- 风险：极低。搜索不改代码，最坏结果是参数没改善，维持现状。
- 收益：高。210 组合在 1 秒内遍历完毕，直接得到全局最优。
- 这是 baseline，应该首先完成。

**扩大搜索到所有参数 — 低风险/低收益**

- 风险：低。仍然是纯搜索。
- 收益：低。固定参数（BETA=2.2, BETA_POS=1.15, LA=1.02 等）是基于金融第一性原理设定的，且 calibration_path_analysis.md 已论证"路径 B（公式驱动）"的合理性。这些参数的物理意义明确，盲目搜索可能破坏公式的理论一致性。
- 只在 3 参数搜索结果不满意时考虑。

**AI 自主决定搜索哪些参数 — 高风险/低收益**

- 风险：高。AI 可能选择搜索 LA（损失厌恶系数）或 BETA（negative certainty 放大），这些参数的值域约束来自行为金融学（如 Kahneman-Tversky 的损失厌恶比率约 2.0-2.5），不应被数据拟合覆盖。
- 收益：低。搜索空间增大但真正的自由度没增加（理论约束已经限定了合理范围）。
- 除非明确给出哪些参数可以搜索、哪些是硬约束，否则不应授权。

**AI 修改单个子项（如 opp_penalty 计算方式）— 高风险/高收益**

- 风险：高。任何结构修改都可能破坏公式的整体平衡（如 opp_penalty 从线性改为非线性可能影响 certainty 的有效范围）。需要人类审核。
- 收益：高（如果修改方向正确）。如果诊断确实发现某个子项是系统性残差的根源，定向修改的边际收益远超参数调优。
- 应该是 Level 2 诊断的后续动作，不应在没有诊断的情况下盲目进行。

**AI 重写整个 _summarize 方法 — 高风险/低收益**

- 风险：极高。`_summarize` 的三分支设计、scarcity 保护、冲突/共识 neutral 区分都是经过深思熟虑的业务逻辑（见 PRD 和 sentiment_score_design.md）。AI 重写大概率会丢失这些隐含约束。
- 收益：低。即使 AI 能找到数值上拟合更好的公式，如果失去了可解释性和理论支撑，对选股决策的可信度反而下降。
- **红线：不应允许。**

## 五、最优校准方案建议

### 方案：分层执行（传统搜索 + AI 诊断，各司其职）

#### Phase 1: 传统工具做参数搜索

```
工具: Python 脚本（网格搜索）
搜索空间: _K x _GAMMA x _OPP_NEG（210 组合）
目标函数: -sent_match * 1000 + MAD * 100 + severe * 10
时间: < 1 秒
输出: top-5 参数组合 + 全场景对比表
```

这一步不需要 Claude Code。写一个 50 行的 Python 脚本（复制 _summarize 逻辑 + 遍历网格 + 排序），执行效率比 ralph-loop 高 3-4 个数量级。

#### Phase 2: Claude Code 做结构诊断

```
工具: Claude Code（单次分析，非 ralph-loop 迭代）
输入: Phase 1 的全场景对比表（特别是残差最大的 5-10 个场景）
任务: 分析残差模式，判断是参数问题还是结构问题
输出: 诊断报告（Option B）
```

**给 AI 的 prompt 要点：**
- 提供 Phase 1 的完整对比表 + top-5 参数组合的结果
- 要求分析残差最大场景的共性特征（都涉及 extreme？都是冲突型？）
- 要求判断：残差是否可以通过参数调优消除，还是公式结构的固有限制
- 不要求 AI 修改代码，只要求诊断

#### Phase 3（条件触发）: 人类决策是否修改公式

如果 Phase 2 诊断发现结构性问题，由人类判断是否需要进入 Level 3。如果需要，再使用 Claude Code（此时 ralph-loop 才有价值）进行定向修改 + 验证循环。

### 自由度、成功定义、红线

**自由度：**
- Phase 1: 零自由度。搜索空间、目标函数、输出格式全部预定义。
- Phase 2: 中等自由度。诊断方向不预设，AI 自主决定关注哪些场景和哪些公式子项。但明确限制为"只诊断不修改"。
- Phase 3: 受控自由度。人类指定修改方向（如"改进 opp_penalty"），AI 在这个约束内自由实现。

**成功定义：**
- Phase 1 成功: sent_match = 100% 且 MAD < 0.05（与现有标准一致）
- Phase 2 成功: 输出一份能回答"需要改公式吗"这个 yes/no 问题的报告
- Phase 3 成功: 修改后的公式在合成场景和 5-10 只真实股票上均表现良好

**红线：**
1. AI 不得自主修改 `_summarize` 的分支结构（三分支 → 两分支、增加新分支等）
2. AI 不得修改 IMPACT_MAP 的映射值（这是输入层的定义，不是聚合层的参数）
3. AI 不得搜索行为金融学锚定的参数（LA, BETA），除非人类明确授权并给出搜索范围
4. AI 不得在没有人类确认的情况下将公式修改写入 `analyzer.py`

## 六、对现有 calibrate_prompt.md 的评估

当前的 `calibrate_prompt.md` 存在三个设计问题：

### 问题 1: 工具选择错位

用 ralph-loop（LLM 迭代）做网格搜索（纯数值计算），是用昂贵的通用工具做廉价的专用工具能完成的事。

**类比：** 用 ChatGPT 逐个计算 210 组 (K, GAMMA, OPP_NEG) 的 MAD，等同于用自然语言描述矩阵乘法然后让 LLM "心算"——理论上能做，实践上荒谬。

### 问题 2: 浪费了 ralph-loop 的独特能力

prompt 中明确规定了搜索参数、范围、步长、终止条件，不留任何"自主决策"空间。ralph-loop 的每轮"自主决定做什么"能力完全被禁锢——它在每轮做的事情都是预定义的（计算一组参数 → 报告 → 下一组）。

**这本质上是把 LLM 当 for 循环使用。**

### 问题 3: 隐含的精度风险

`calibrate_prompt.md` 要求 AI "精确复制 _summarize 逻辑"进行计算。但 LLM 做浮点运算存在以下风险：
- `math.exp(-evidence / _K)` 在 evidence 接近 0 时的精度
- 多步乘法的误差累积
- 条件分支（if/elif/else）在边界值附近的判断稳定性

这些在 Python 脚本中是零成本零风险的，但在 LLM 中是真实的精度隐患。

### 改进建议

将 `calibrate_prompt.md` 拆分为两个独立步骤：

1. **`calibrate_script.py`**: 50 行 Python 脚本，做纯网格搜索，输出全场景对比表。
2. **`diagnose_prompt.md`**: Claude Code 分析 prompt，输入对比表，输出诊断报告。

这样每个工具都用在了其最擅长的地方。

## 七、与项目现有基础设施的关系

项目中 `BreakoutStrategy/mining/threshold_optimizer.py` 已有成熟的搜索框架：

| 能力 | threshold_optimizer.py | 当前校准需求 |
|------|----------------------|-------------|
| 搜索算法 | Beam Search + Optuna TPE | 网格搜索足够（仅 210 组合） |
| 目标函数 | shrinkage score | MAD + sent_match |
| Checkpoint | pickle 续跑 | 不需要（< 1 秒完成） |
| 搜索空间定义 | quantile-based bounds | 手动指定范围 |

当前任务的搜索空间（3 参数 x 210 组合）甚至不需要 Optuna。一个简单的三重 for 循环就足够了。Optuna TPE 适合的是高维搜索空间（threshold_optimizer 中 12 个因子 x 连续值域），当前任务远未达到这个复杂度。

## 八、总结：每个工具的甜区

| 工具 | 甜区 | 当前任务中的角色 |
|------|------|----------------|
| **Python 脚本 (for 循环)** | 小搜索空间的穷举（< 1000 组合） | **Phase 1 主力** |
| **Optuna / scipy** | 高维连续搜索空间、需要自适应采样 | 当前不需要 |
| **Claude Code (单次分析)** | 理解代码语义、残差模式分析、因果推理 | **Phase 2 主力** |
| **Claude Code (ralph-loop)** | 需要多轮"分析→修改→验证"循环的开放式任务 | **Phase 3 备选** |

**最终建议的一句话总结：** 把网格搜索从 ralph-loop 中解放出来交给 Python 脚本，把 Claude Code 的能力集中在它真正擅长的结构诊断上。
