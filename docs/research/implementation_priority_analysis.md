## 一、问题全景

当前系统存在 5 个独立子问题:

| ID | 问题 | 严重度 | 典型案例 |
|----|------|--------|---------|
| **P1** | amplitude 度量混淆振荡与超涨 | 中 | 低价股高振荡被误标 turbulent |
| **P2** | 固定阈值对低价股系统性不公平 | 中 | $1-$10 误杀率 32% vs $50+ 仅 3% |
| **P3** | 二值 turbulent 的悬崖效应 | 低-中 | amp=0.79 全保留 vs 0.81 全清零 |
| **P4** | 信号能量耗散未被追踪 | **高** | VTGN: weighted_sum=9.0 但已完成派发 |
| **P5** | 缺乏派发完成的宏观识别 | 低 | 用户无法一眼看出"这只股票已派发完成" |

**用户当前最痛的问题是 P4** — 排序结果中高分股票的走势已经结束，直接影响选股质量。

## 二、候选方案全景

| 方案 | 来源 | 解决问题 | 实现成本 | 独立可用 |
|------|------|----------|---------|---------|
| A: 波动率自适应 | turbulent 报告 | (P2) | 中 | **否** (效果倒退) |

| L2: 派发完成检测 | freshness 报告 | P5 | 低 | 有条件 |

## 三、交叉审查关键发现

### 3.1 方案 A 应排除

波动率自适应与用户场景根本矛盾。用户目标群体（$1-$10 低价股）天然高波动，用自身波动率标准化等于"因为它一直波动所以暴涨正常"。78.8% 的目标股逃脱过滤。**不应实施。**

### 3.2 D_grad 与 L1 高度重叠

两者都在降低 BVY 信号的 effective_strength，但粒度不同:
- D_grad: 股票级统一乘以 decay_weight（基于整体振幅）
- L1: 信号级独立乘以 freshness（基于每个信号的价格轨迹）

**如果同时实现会产生双重惩罚**:
```
effective = strength × freshness(L1) × decay_weight(D_grad)
```
一个 freshness=0.5 的信号（仍有 50% 能量）会被 decay_weight=0.28 压到 0.14，过度惩罚。

**结论**: L1 在信息量和精确度上严格优于 D_grad。当 L1 存在时，D_grad 的边际价值极低。**二选一，选 L1。**

### 3.3 L1 可近似替代 turbulent 的二值机制

| 场景 | turbulent 行为 | L1 行为 | 等价？ |
|------|---------------|---------|--------|
| 暴涨暴跌 (VTGN) | BVY=0, D=full | BVY freshness≈0, D≈保留 | 是 |
| amp=0.79 (刚低于阈值) | BVY=full | BVY 按 PRTR 衰减 | **L1 更精确** |
| 极端暴涨 + 新信号 | BVY=0 (误杀新信号) | 新信号 freshness=0.9 保留 | **L1 更准确** |

L1 让系统对 turbulent 二值判断的依赖大幅降低，但建议保留 turbulent 作为极端情况的安全兜底。

### 3.4 C 是 B_pre 的强前置

分层阈值应基于 max_runup（而非 amplitude）计算 P95。如果先实现 B_pre 再换 C，阈值需全部重算。**必须先 C 后 B_pre。**

### 3.5 B_pre "退化为分段阈值" 是优点不是缺点

审查结论：确定性、可解释性、可调性、零运行时成本。"数据驱动的分段固定阈值" 比"拍脑袋的全局固定阈值"严格更好。

## 四、最终优先级列表

### P0: 立即实现 — C (max_runup 替代 amplitude)

| 维度 | 评估 |
|------|------|
| **解决问题** | P1: amplitude 混淆"高振荡"和"方向性超涨" |
| **改动量** | 替换 `calculate_amplitude()` 函数体，~5 行代码 |
| **新增参数** | 0 |
| **风险** | 极低 — API 签名不变，返回值语义兼容，可沿用现有阈值 0.8 |
| **ROI** | **最高** — 每行代码产出的收益最大 |
| **为什么第一个做** | 所有后续 turbulent 改进的基础；语义修正是最底层的改动 |

**预期效果**: 减少 ~16% 的 turbulent false positive（55 只仅高振荡的低价股不再被误标）。

---

### P1: 立即实现 — L1 (PRTR 信号鲜度)

| 维度 | 评估 |
|------|------|
| **解决问题** | **P4: 信号能量耗散未追踪**（用户最痛的问题）；间接缓解 P3 |
| **改动量** | ~80-100 行新函数 + scanner worker 中增加鲜度计算 |
| **新增参数** | 2 个 (alpha=1.5, half_life=30) |
| **数据流变更** | **极小** — 在 worker 内计算，直接修改 signal.strength，无额外 IPC |
| **风险** | 低 — 向后兼容；small-rise 敏感性可用 3 行 min_rise_pct 守卫修复 |
| **ROI** | **高** — 排序系统最大的信息缺失得到修复 |
| **为什么紧跟 C** | C 修正度量，L1 修正计分。两者完全正交，组合覆盖 P1+P4 |

**预期效果**: VTGN 类案例 weighted_sum 从 9.0 降至 ~2.2。已完成派发的股票大幅降低排名。

**实现要点**:
- 在 `scan_single_stock()` 返回前计算 freshness 并乘入 `signal.strength`
- Worker 已有 `df_slice`、`filtered_signals`、`scan_date`，无需额外数据
- Aggregator 无需任何改动（看到的是已调整的 strength 值）
- 增加 `min_rise_pct=0.05` 守卫解决 small-rise 边界

---

### P2: 建议实现 — B_pre (分层 P95 阈值)

| 维度 | 评估 |
|------|------|
| **解决问题** | P2: 固定阈值对低价股系统性不公平 |
| **改动量** | 离线预计算脚本 + 配置文件新增 tier 阈值 + aggregator 查表逻辑 |
| **新增参数** | 2 个 tier 边界 ($5, $20) + 3 个自动计算的 P95 阈值 |
| **前置依赖** | **C 必须先完成** — 分层阈值应基于 max_runup 计算 |
| **风险** | 中 — 需要跑全量数据验证；$0-2 和 $2-5 可能需要进一步细分 |
| **为什么排第三** | 解决的是公平性问题，不如 P4(排序准确性) 紧迫；且依赖 C |

**预期效果**: 各价位 turbulent 过滤率趋于一致 (~5%)，消除 10 倍误杀比。

---

### P3: 可选实现 — L2 (派发完成检测)

| 维度 | 评估 |
|------|------|
| **解决问题** | P5: 缺乏派发完成的宏观识别 |
| **改动量** | ~30 行检测函数 + SignalStats 新增 bool 字段 + UI 集成 |
| **新增参数** | 3 个阈值 (peak_recency=0.60, drawdown=0.35, position=0.30) |
| **独立价值** | **中等** — 纯 UI 标记，不改变排序。L1 已降低排名后，L2 锦上添花 |
| **为什么排第四** | 如果 L1 已将 VTGN 推到列表底部，用户可能根本看不到它，L2 标记的必要性降低 |

**预期效果**: 用户在列表中看到"派发完成"标记，可快速跳过或过滤。

---

### 延后/跳过: D_grad (渐进降权)

| 维度 | 评估 |
|------|------|
| **解决问题** | P3: 二值 turbulent 悬崖效应 |
| **为什么延后** | L1 已通过逐信号精确衰减间接消除了对二值判断的严重依赖；D_grad 与 L1 同时实现会产生双重惩罚；边际价值极低 |
| **何时有价值** | 仅当决定不实现 L1 时，D_grad 才成为 P3 的主要解法 |

---

### 排除: A (波动率自适应)

**不应实施。** 方法论与用户场景根本矛盾。

```
Phase 0 (P0): C — max_runup 替代 amplitude
  ├─ 改动: composite.py (5 行)
  ├─ 测试: 对比替换前后标记结果
  └─ 独立可用: ✓

Phase 1 (P1): L1 — PRTR 信号鲜度
  ├─ 改动: composite.py (~80 行新函数), scanner.py (worker 内计算)

  ├─ 改动: composite.py (30 行), models.py, scanner.py, UI
  ├─ 测试: 场景验证 (VTGN 类 / 正在拉升 / 底部蓄力)

```

## 七、关键结论

1. **L1 (PRTR) 是整个改进计划中 ROI 最高的单项**。它直接解决用户最痛的问题 (P4)，且间接缓解 P3。如果只能做一件事，做 L1。

2. **C (max_runup) 成本极低 (5 行)，应作为第一步**。它是后续所有 turbulent 改进的基础，且即刻产生价值。

3. **D_grad 在 L1 存在时不应实施**。两者功能重叠，同时存在会双重惩罚。如果未来决定不要 L1，D_grad 可以复活。

4. **A (波动率) 永远不应实施**。方法论与场景矛盾。

5. **Phase 0 + Phase 1 是最小有效投资**。两者正交、无冲突、合计覆盖 4/5 个问题。
