# 阻力属性因子审核报告

> 审核日期: 2026-03-18
> 审核范围: age, test, height, peak_vol 四个阻力属性因子
> 审核依据: 技术分析理论、市场微观结构、因子设计合理性

---

## Executive Summary

四个阻力属性因子整体设计思路正确——它们从不同维度量化了被突破阻力位的"重要程度"，阻力位越重要，突破后的上涨动能越强。这一基本假设在技术分析理论中有坚实的理论支撑。

然而，在具体实现和参数设定上，存在以下核心问题：

1. **age 因子奖励值过于保守**，与其金融学意义不匹配，且最低阈值 42 天偏低
2. **test 因子聚类算法存在结构性缺陷**，3% 固定阈值在不同价格波动率环境下表现不一致
3. **height 因子奖励值跨度过大**（1.3 到 2.0），且阈值 0.7（70%）在实际市场中的触发条件需要审视
4. **peak_vol 因子阈值偏高**，可能遗漏大量有意义的放量形态

---

## 因子 1: age（突破位龄）

### 1.1 设计意图分析

**核心假设**: 越老的阻力位被突破，意味着更长期的供给被消化完毕，突破后上涨空间更大。

**金融学依据**:
- 老的阻力位代表历史上更多投资者在该价位建仓/被套。时间越久，"套牢盘"中一部分已经止损离场（供给逐渐被吸收），但市场记忆仍在——当价格再次触及该位时，剩余持仓者的行为（继续持有 vs 卖出）成为关键。
- 从行为金融学角度，时间久远的阻力位突破更具有"心理突破"意义：长期盘整后的突破往往伴随着市场参与者预期的重新校准（re-anchoring），容易形成新的上升趋势。
- William O'Neil 的 CANSLIM 系统和 Stan Weinstein 的阶段分析都强调"基底时间越长，突破后涨幅越大"（the longer the base, the higher in space）。

**结论**: 设计意图合理，理论支撑充分。

### 1.2 计算逻辑审核

```python
age = max(idx - p.index for p in broken_peaks)
```

**逻辑评价**: 取最老被突破峰值的年龄。逻辑简洁正确。

**潜在问题**:
- **取 max 的合理性**: 当一次突破同时突破多个峰值时，取最老峰值的年龄。这是合理的——最老的阻力位代表最长期的供需平衡被打破，是突破重要性的上限衡量。
- **无上限截断**: 极端情况下 age 可能非常大（如 500+ 天），但由于奖励值是阶梯式而非线性，不会造成分数爆炸，设计合理。

### 1.3 参数审核

| 阈值 | 含义 | 奖励值 | 评价 |
|------|------|--------|------|
| 42 天 | ~2个月 | 1.02 | 阈值偏低，奖励偏弱 |
| 63 天 | ~3个月/一个季度 | 1.03 | 合理基准 |
| 252 天 | ~1年 | 1.05 | 奖励偏弱 |

**问题与建议**:

1. **奖励值过于保守**: age 因子的最大奖励仅为 1.05（5%），这与其金融学意义严重不匹配。突破一年期阻力位是非常重大的技术信号，仅 5% 的奖励无法体现其重要性。相比之下，height 因子最高可达 2.0（100% 奖励），test 因子最高 1.4（40% 奖励）。建议将奖励值提升至 (1.05, 1.10, 1.20) 范围。

2. **最低阈值 42 天偏低**: 42 个交易日约 2 个自然月。从技术分析角度，一个有意义的"老"阻力位至少应该是一个季度以上。2 个月的阻力位在大多数交易者的记忆中尚属"近期"，其突破不应获得额外奖励。建议将最低阈值调整为 63 天（一个季度），三级阈值改为 (63, 126, 252)。

3. **缺少超长期阈值**: 可以考虑增加一个 504 天（~2年）的阈值档位，因为突破两年以上的阻力位是极其罕见且有力的信号。

### 1.4 边界条件

- **数据长度限制**: 如果股票历史数据不足一年，age 不可能达到 252 天的阈值。这不是 bug，但意味着对短历史数据的股票，该因子区分力不足。可接受。
- **broken_peaks 为空**: 代码上游保证 broken_peaks 非空（有突破必有被突破峰值），无问题。

## 因子 2: test（测试次数）

### 2.1 设计意图分析

**核心假设**: 某价格区域被多次测试（形成多个峰值）后突破，说明该阻力位经过反复确认，突破后更有意义。

**金融学依据**:
- **多次测试 = 供给充分吸收**: 每次价格触及阻力位后回落，一部分卖方在该价位完成了出货。经过多次测试，该价位的潜在抛压逐渐耗尽。当最终突破发生时，头顶已无悬挂供给，上涨阻力大幅减小。
- **经典技术分析中的"triple top breakout"**: 三重顶突破被认为是极强信号，原因正是三次测试充分暴露了该价位的供需关系。Edwards & Magee 的《Technical Analysis of Stock Trends》对此有详细论述。
- **支撑/阻力的"翻转"效应**: 被多次测试后最终突破的阻力位，会变成更强的支撑位（角色互换原则），这为突破后的持续上涨提供了"底部保护"。

**结论**: 设计意图非常合理，是最具经典技术分析理论支撑的因子之一。

### 2.2 计算逻辑审核

```python
def _calculate_test(self, broken_peaks) -> int:
    sorted_prices = sorted(p.price for p in broken_peaks)
    best_cluster = 1
    current_cluster = 1
    for i in range(1, len(sorted_prices)):
        if (sorted_prices[i] - sorted_prices[i-1]) / sorted_prices[i-1] <= self.cluster_density_threshold:
            current_cluster += 1
        else:
            best_cluster = max(best_cluster, current_cluster)
            current_cluster = 1
    return max(best_cluster, current_cluster)
```

**聚类阈值**: `cluster_density_threshold` 默认为 `peak_supersede_threshold`（3%）。

**逻辑评价**: 贪心聚类算法本身正确——按价格排序后，相邻价格差 ≤ 3% 归为同一簇。

**潜在问题**:

1. **3% 固定阈值的波动率偏差**: 对于低波动股票（如公用事业股，年化波动率 15%），3% 的价格差已经相当大，可能将本不属于同一阻力区的峰值错误归类。对于高波动股票（如成长股，年化波动率 50%），3% 又太小，可能将同一阻力区的峰值拆分成多个簇。

   **建议**: 考虑使用波动率自适应阈值，例如 `threshold = k * daily_vol * sqrt(T)`，其中 T 为平均测试间隔天数。但这会增加复杂度。当前 3% 阈值与 `peak_supersede_threshold` 保持一致的设计有其简洁性优势，作为 v1 可以接受，但需要意识到这一局限性。

2. **贪心聚类的链式漂移问题**: 考虑三个价格 100, 102.9, 105.8。100→102.9 差 2.9%（≤3%），102.9→105.8 差 2.8%（≤3%），因此三者被归为一簇。但 100→105.8 差 5.8%，在实际意义上可能不属于同一阻力区。这种"链式漂移"在峰值数量较多时可能导致过大的簇。

   **建议**: 可以增加一个簇内最大价格跨度限制（如簇内最高价/最低价 ≤ 1.05），防止链式漂移。但考虑到 broken_peaks 数量通常较少（2-5 个），此问题在实践中影响有限。

3. **只统计 broken_peaks**: 当前只统计"被突破的峰值"中的簇，而不是"所有曾在该价位出现过的峰值"。这意味着如果一个阻力区有 5 个历史峰值，但突破时只有 3 个仍然活跃（另 2 个已被新峰值 supersede），test 值最大只能是 3。这个行为是否符合设计意图需要确认——从"该阻力区被测试了多少次"的角度看，应该统计所有历史峰值；从"当前突破消化了多少阻力"的角度看，只统计 broken_peaks 是合理的。当前实现偏向后者。

### 2.3 参数审核

| 阈值 | 含义 | 奖励值 | 评价 |
|------|------|--------|------|
| 2 次 | 双顶突破 | 1.10 | 合理 |
| 3 次 | 三顶突破 | 1.25 | 合理 |
| 4 次 | 四顶及以上 | 1.40 | 合理 |

**评价**: 阈值分级合理。双顶是最基本的有意义阻力形态（1 次测试没有形成"阻力区"的含义），三顶是经典强阻力形态，四顶及以上在实践中较少见但信号极强。奖励值的递增幅度（+10%, +15%, +15%）也合理，体现了边际递增效应。

**唯一顾虑**: test ≥ 4 的情况实际触发频率可能较低，需要确认是否有足够样本支撑 1.4 的奖励值。但从纯理论角度，四次以上测试的阻力位突破确实是极强信号，1.4 的奖励值不算过高。

## 因子 3: height（峰值高度）

### 3.1 设计意图分析

**核心假设**: 峰值越"突出"（relative_height 越大），该阻力位越显著，突破后意义越大。

**金融学依据**:
- **显著峰值 = 明确的供需转折点**: 一个高 relative_height 的峰值意味着价格在该点达到了相对于周围区间的显著高点，然后大幅回落。这表明在该价位存在强劲的卖方力量。
- **视觉显著性 → 市场关注度**: 从市场微观结构角度，越显著的峰值越容易被技术分析交易者识别并标注为阻力位。当价格再次接近该位时，这些交易者会集中在此设置止盈/止损单，形成自我实现的预言。突破这些"众所周知"的阻力位会触发大量买方订单（空头回补 + 突破追涨），推动价格进一步上涨。
- **V 型反转底部的深度**: relative_height 本质上衡量的是峰值到局部谷底的深度。深度越大，代表从峰值到谷底的价格下跌幅度越大，从而突破该峰值意味着从更深的底部反弹上来，动量更强。

**结论**: 设计意图合理。

### 3.2 计算逻辑审核

```python
# _create_peak 中的 relative_height 计算
side_bars = self.total_window // 2  # 默认 10 // 2 = 5
left_start = max(0, idx - side_bars)
right_end = min(current_idx, idx + side_bars + 1)
window_low = min(self.lows[left_start:right_end])
relative_height = (price - window_low) / window_low
```

```python
# _calculate_height
height = max(p.relative_height for p in broken_peaks)
```

**逻辑评价**:

1. **relative_height 定义正确**: (峰值价格 - 局部最低价) / 局部最低价，衡量峰值相对于周围的突出程度。
2. **取 max 合理**: 取所有被突破峰值中最显著的那个，代表突破消化的最大阻力。
3. **对称窗口避免稀释**: 使用 `total_window // 2` 的对称窗口而非全检测窗口，注释明确说明"避免长窗口跨越其他峰值导致 relative_height 被稀释"，设计考虑周到。

**潜在问题**:

1. **窗口大小限制**: `side_bars = total_window // 2 = 5`。这意味着 relative_height 只看峰值左右各 5 根 K 线内的最低价。对于一些宽幅震荡的股票，真正的"谷底"可能在 5 天以外。这会导致 relative_height 被低估。但扩大窗口又会引入跨越其他峰值的稀释问题，这是一个 trade-off，当前选择偏保守但合理。

2. **relative_height 在峰值创建时计算，不会更新**: 如果峰值创建后，其右侧出现更深的谷底，relative_height 不会被更新。但这符合"峰值形成时刻的显著性"语义——我们关注的是峰值在形成时有多显著，而不是后续走势。

### 3.3 参数审核

| 阈值 | 含义 | 奖励值 | 评价 |
|------|------|--------|------|
| 0.2 (20%) | 中等显著性 | 1.30 | **起步奖励偏高** |
| 0.4 (40%) | 高显著性 | 1.60 | 合理偏高 |
| 0.7 (70%) | 极高显著性 | 2.00 | **可能过高** |

**问题与建议**:

1. **奖励值跨度过大**: 从 1.3 到 2.0，最大奖励值是基准分的 2 倍。在乘法模型中，这意味着 height 因子单独就能让总分翻倍。与其他因子相比（age 最大 1.05，test 最大 1.4），height 的权重显得过于突出。需要问：height 真的比 test（多次测试确认的阻力区突破）重要两倍以上吗？

   **建议**: 考虑将奖励值压缩至 (1.15, 1.30, 1.50) 范围。一个 relative_height=0.7 的峰值确实非常显著，但 2.0 的乘数在乘法模型中影响过大。

2. **阈值 0.7 (70%) 的触发条件审视**: 对于一只 $100 的股票，relative_height=0.7 意味着峰值周围 5 天内存在 $58.8 的低价（100/1.7≈58.8）。这在日常交易中是极端情况——通常只有经历过大幅暴跌后的反弹才会出现。但需要注意：
   - 窗口仅 5 天，如果一只股票在 5 天内从 $59 涨到 $100，这代表 ~70% 的涨幅，属于极端走势（可能是低价股或SPAC等特殊标的）。
   - 对于这类极端 relative_height，给予最高奖励需要谨慎——它可能代表的是投机性暴涨而非健康的技术突破。

3. **最低阈值 0.2 (20%) 分析**: 5 天内从底到顶 20% 的波动，对大多数正常股票来说已经较为显著。作为起步奖励阈值基本合理。

4. **与 min_relative_height 的关系**: 检测器的 `min_relative_height` 默认为 0.05 (5%)，即 relative_height < 5% 的峰值不会被检测到。因此 height 因子的实际输入范围是 [0.05, +∞)。0.2 的第一级阈值留出了足够的中间空间（0.05-0.2），这部分峰值被认为"不够显著，不给奖励"，合理。

### 3.4 重要风险

**波动率偏差问题**: relative_height 是绝对百分比，不是波动率标准化的。高波动股票天然更容易产生高 relative_height（因为日内波动大，5 天内的最高最低差就大），而低波动股票几乎不可能达到 0.4 或 0.7 的 relative_height。这意味着 height 因子对高波动股票系统性地给予更高奖励，对低波动股票系统性地偏低。

**这是否是问题取决于设计意图**: 如果认为"高波动股票的显著峰值突破本身就更有力"，则当前设计可接受。但如果希望 height 因子在不同波动率的股票之间具有可比性，应考虑使用 ATR 标准化的 relative_height。

## 因子 4: peak_vol（峰值量能）

### 4.1 设计意图分析

**核心假设**: 峰值形成时放量越大，说明在该价位有大量交易发生，该阻力位越"重要"。

**金融学依据**:
- **放量 = 大量仓位建立**: 峰值日大放量意味着大量买方在顶部接盘，这些仓位在随后的价格下跌中变成套牢盘。当价格再次上涨到该位置时，这些套牢者有强烈的"解套出货"冲动，形成真实的卖压。
- **突破放量阻力位的含义**: 如果一个峰值形成时交易量很大（意味着该价位有大量持仓成本），那么价格能够突破该位，说明市场买方力量足以消化所有这些潜在卖压，是极强的看涨信号。
- **Volume Profile 理论**: 高成交量价位（High Volume Node）被认为是市场的"公允价值区域"。突破 HVN 意味着市场共识发生了根本性转变。
- **经典的量价关系理论**: "量在价先"（Volume precedes price）——有大量成交量支撑的价格行为更具持续性。

**结论**: 设计意图非常合理，有充分的理论和实践支撑。

### 4.2 计算逻辑审核

```python
# Peak 创建时
window_start = max(0, idx - 63)
avg_volume = np.mean(self.volumes[window_start:idx])
vol_ratio = self.volumes[idx] / avg_volume

# _calculate_peak_vol
peak_vol = max(p.volume_peak for p in broken_peaks)
```

**逻辑评价**:
1. **63 天均量基准**: 使用 3 个月均量作为基准计算放量倍数，与突破量能因子（volume）的基准一致，合理。
2. **取 max**: 取所有被突破峰值中最大的放量倍数。逻辑正确——关注最显著的量能信号。

**潜在问题**:

1. **峰值日放量 vs 峰值区域放量**: 当前只看峰值当天的成交量。但在实际市场中，一个重要的阻力位通常不是单日形成的——多日持续放量形成的"量能堆积"比单日天量更有意义。例如，一只股票在 $50 附近连续 5 天放量（每天 2x 均量），总换手远大于某天 5x 放量。建议考虑使用峰值周围若干天的平均放量倍数，而非单日值。但这会增加实现复杂度，且与 Peak 数据结构的设计不完全匹配。作为 v1，当前实现可接受。

2. **均量基准的窗口起点问题**: `window_start = max(0, idx - 63)`。如果股票数据不足 63 天，均量基准可能不稳定（少量样本）。但 `max(0, ...)` 保证了安全性，且数据不足时 `avg_volume` 会偏高（少量高成交量日不被稀释），导致 vol_ratio 偏低——这是保守的偏差，可接受。

### 4.3 参数审核

| 阈值 | 含义 | 奖励值 | 评价 |
|------|------|--------|------|
| 3.0 倍 | 中等放量 | 1.10 | **阈值偏高** |
| 5.0 倍 | 大幅放量 | 1.20 | 合理 |

**问题与建议**:

1. **最低阈值 3.0 倍偏高**: 在美股市场中，3 倍均量已经是相当显著的放量。许多有意义的峰值伴随着 1.5-3.0 倍的放量，但当前设计完全忽略了这个区间。建议增加一个 2.0 倍的入门阈值。

   **建议阈值**: (2.0, 3.0, 5.0)，对应奖励值 (1.05, 1.10, 1.20)。

2. **只有两级阈值**: 相比其他因子（age 3 级、test 3 级、height 3 级），peak_vol 只有 2 级。这导致其区分力不够精细。增加一个入门阈值可以改善这一点。

3. **奖励值整体偏低**: 峰值量能是一个非常有意义的阻力位重要性指标。最高奖励 1.2 与 age（最高 1.05）相比合理，但与 height（最高 2.0）和 test（最高 1.4）相比偏低。考虑到"重量级阻力位突破"的重要性，5 倍放量峰值的奖励可以适当提高至 1.25-1.30。

### 4.4 边界条件

- **vol_ratio 为负或零**: `avg_volume > 0` 的检查保证了不会出现除零错误。如果 avg_volume=0（所有历史成交量为零），vol_ratio 返回 1.0，不会触发奖励。合理。
- **数据源问题**: 某些数据源可能包含异常成交量数据（拆股日、停牌恢复日等），这可能导致 vol_ratio 异常偏高。但这是数据层面的问题，不属于因子逻辑问题。

## 跨因子分析

### 因子间权重平衡

在乘法模型中，各因子的最大奖励值直接决定了其对总分的影响权重：

| 因子 | 最大奖励 | 最大贡献(%) | 评价 |
|------|---------|------------|------|
| age | 1.05 | +5% | 过低 |
| test | 1.40 | +40% | 合理 |
| height | 2.00 | +100% | 过高 |
| peak_vol | 1.20 | +20% | 偏低 |

**问题**: height 因子的影响力远超其他三个因子之和。在乘法模型中，如果 height=2.0 且 test=1.4 同时触发，仅阻力属性就能将分数提升 2.8 倍（乘数 = 2.0 × 1.4 = 2.8）。这是否符合设计意图？

**建议的平衡方案**:

重新考虑四个因子的相对权重，使其反映各自在预测突破后上涨中的相对重要性。从技术分析理论来看，test（多次测试确认）和 height（阻力位显著性）应该是最重要的两个因子，peak_vol 次之，age 再次之。建议的最大奖励值比例约为：

- age: 1.15 (中等重要性)
- test: 1.40 (保持不变，高重要性)
- height: 1.50 (降低，高重要性但不应独自主导)
- peak_vol: 1.30 (提升，中高重要性)

### 因子间相关性

需要注意的潜在相关性：

1. **age 与 test**: 可能存在正相关——老的阻力位有更多时间积累测试次数。但这不是必然的（一个老的峰值可能只被测试过一次）。轻度正相关可接受。

2. **height 与 peak_vol**: 可能存在正相关——高显著性的峰值（大涨大跌）通常伴随大成交量。这意味着两个因子在乘法模型中可能会对同一类突破事件"双重奖励"。当前 peak_vol 的奖励值较低（最大 1.2），部分缓解了这个问题。

3. **height 的波动率偏差**: 如前所述，height 天然偏向高波动股票。如果系统的目标是在不同波动率的股票之间公平比较突破质量，这是一个需要关注的系统性偏差。

## 综合建议

### 优先级排序

1. **[高]** 重新平衡 height 因子的奖励值（从 2.0 降至 ~1.5），避免其独自主导评分
2. **[高]** 提升 age 因子的奖励值（从 1.05 到 ~1.15），使其影响力与金融学意义匹配
3. **[中]** 为 peak_vol 增加 2.0 倍的入门阈值，提升覆盖率
4. **[中]** 考虑 test 因子聚类算法的链式漂移防护（增加簇内最大跨度限制）
5. **[低]** 考虑 height 因子的波动率标准化（需权衡增加的复杂度）
6. **[低]** age 因子最低阈值从 42 天调整至 63 天

### 整体评价

四个阻力属性因子的**设计方向完全正确**，从多个维度（时间、测试次数、视觉显著性、量能）量化了阻力位的重要程度。金融学理论支撑充分。主要问题集中在**参数平衡**上——height 因子权重过大，age 因子权重过小，peak_vol 阈值起点偏高。通过参数调整即可优化，不需要修改核心算法逻辑。
