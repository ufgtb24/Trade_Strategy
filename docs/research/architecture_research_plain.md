## 一、我们在解决什么问题?

### 现状打个比方

你现在的选股系统像是这样:

> **每次股票"突破"重要阻力位的那一根 K 线,系统就拍一张照片**。
> 照片里记录 11 个数字(成交量、突破力度、突破前的均线位置等等),叫"因子"。
> 系统用历史数据挖掘,找出"哪些数字组合的照片对应着后续大涨" — 这就是一条"模板"。
> 实盘时,新照片被拍下来后,跟模板对一对,匹配的就是好股票。

这套系统叫**BO 因子框架**(BO = Breakout = 突破)。它在做"**离线挖掘 + 评分规则生成**":挖好规则,实盘按规则打分排序。

### 用户提出的难题

某天你看图发现一种走势:

1. **短期内连续好几次突破**(不是一次,是一串)
2. **每次都放量**
3. **突破之前 MA40 几乎是水平的**(说明此前股价"睡大觉"很久了)
4. **最后一次突破之后,股价稳定在比突破前更高的位置**(形成一个台阶)

这条规律有点麻烦:它不是关于**一次**突破的,而是关于**一连串现象**的;还有顺序要求(MA40 平 → 突破 → 台阶);更要命的是 — **特征 4 的"台阶"是在最后那次突破之**后**的事**,系统拍照那一刻还看不到。

**用户的问题**:这种规律我现在的系统能描述吗?要不要换架构?

## 二、我们考虑了哪几条路?

### 路径 A:把当前架构升级成 Condition_Ind 风格

**Condition_Ind** 是隔壁项目 `new_trade` 的一个机制。比喻:

> 它像一串"过期通知" — 每个条件像一个有效期标签:
> "MA40 水平这件事,过去 40 天里发生了"(有效期 40 天,如果在期内 → 仍算数)
> "放量这件事,过去 10 天里发生了"(过期了 → 不算)
> 几个标签都"还在期"的某根 K 线,就发出一个 valid=True 信号。

它能做现在系统做不到的事:**讲顺序**("先 X,再 Y")、**讲嵌套**(一个条件本身可以是另一组条件的组合)。

### 路径 B:不动架构,只加一个新因子

**比喻**:

> 当前系统不是已经能在"BO 那一刻拍照"吗?那我就**多拍一个数字**进照片就行 — 比如再加一个数字 `ma_flat`,意思是"突破前 40 天 MA40 的水平度"。
> 这样规律里的 1+2+3 都已经在系统能看到的"突破前历史"里了,系统挖矿时自然能挖出"这种照片对应大涨"的模板。

**唯一的代价**:特征 4("事后的台阶")没法直接放进照片 — 因为拍照那一刻台阶还没形成。**但**:既然系统是用"未来涨幅"作为衡量标准来挖好坏模板的,它会**间接偏好**那些"事后真的形成了台阶的"样本(因为没台阶的样本通常涨不上去)。这叫"label 隐含表达"。

### 路径 C:上一套大杀器(MATCH_RECOGNIZE)

**比喻**:

> 它是数据库里一个标准 SQL 子句,专门为"事件序列正则匹配"设计 — 你写 `先一段 FLAT,再 2-5 个 BO,再一段 STEP_UP`,数据库就帮你扫历史找出所有这样的"模式实例"。
> 它能优雅地表达"事后台阶"(因为模式的最后一段就是 STEP_UP)。

但代价大:得自己写一个 pandas 版的解释器、把现有的挖掘流水线推倒重建,工作量 5-7 周。

## 三、第一轮研究的结论

3 位 AI 专家 + 一轮交叉互喷之后,达成共识:

### 结论 1:"事后台阶"是物理问题,不是架构问题

> 你问"明天的彩票号码是多少",换什么架构都答不上来。
> "BO 之后 10 天的台阶"也一样:**实盘那一刻未来还没发生**,你换 MATCH_RECOGNIZE、换 Condition_Ind 都看不见。

MATCH_RECOGNIZE 的 SQL 写起来漂亮,但**实盘执行时还是要等那 10 天过去才能确认信号**。所谓"它能优雅表达事后窗口"只是**表达自然**,**观测仍然要等**。物理时延逃不掉,与架构无关。

### 结论 2:这一条规律,加一个因子就够了(路径 B)

- 规律 1+2+3 → 现有系统加一个 `ma_flat` 因子就能挖矿挖出来
- 规律 4 → 让"未来涨幅最大"的标准间接偏好它,不显式约束
- 工作量 < 1 周
- MATCH_RECOGNIZE 的 5-7 周大动干戈,**单条规律分摊不下来,是 over-engineering**

### 结论 3:什么时候才该上大杀器?

只有当出现以下任一情况:
- 出现 ≥2 条**事后**规律,而且这些事后段彼此关联(需要"STEP 段的 close 与 BO 段的 resistance 比较"这种跨段引用)
- 出现真的需要"先 X 后 Y"强顺序角色的规律(不是聚类、不是计数,是带角色的序列)
- 这类复杂规律累积 ≥5 条

**触发器没勾上之前 → 走路径 B**。

## 四、用户的关键洞察(改了游戏规则)

研究做完之后,用户提出一个很妙的想法:

> "**把'平台形成'本身当成主事件**,BO 反过来作为它的前置条件,然后扫描这个'以 BO 作为前缀的平台形成事件'。"

**比喻**:

> 之前系统是这么干的:每次拍 BO 照片,然后问"未来会不会有台阶"。
> 用户的新想法:**改成在"台阶已经形成"那一刻拍照**,然后问"过去这段时间里有没有出现过 BO?"
>
> 这就把"未来"问题变成了"过去"问题 — **拍照时点向后挪了 K 天**,这 K 天发生的事现在全是历史,完全可见,不存在未来不可观测的问题了。

事后台阶不再是 BO 事件的事后特征,而是 Platform 事件的**定义本身**。任何 Platform 事件**按定义**都已经有台阶 — 这把"形态完整性"从统计偏好(挖矿希望它有)升级成了**硬约束**(没台阶的样本根本不会触发 Platform 事件)。

物理时延仍然在(实盘还是要等 K 天才能确认),但**信号语义从"BO 已发生,未来未知"变成"K 天前的 BO 已经被验证形成台阶,可以买入"** — 这才是用户实际下单的时刻。

## 五、第二轮研究:仔细读 new_trade 的真实代码

第一轮研究只看了 Condition_Ind 的 API 文档,没深入读它在生产中**实际怎么用**。第二轮派人去读了真实代码,发现一些很重要的事:

### 发现 1:Condition_Ind 的"链式 DSL"主要是个错觉

- `Result_ind`(更复杂的版本)**已经被删除**,只剩注释里一句残骸
- `min_score` 这个参数**生产中从未启用**,代码里写着 `# min_score=3` 注释掉
- `keep / keep_prop / relaxed / exp_cond`(更精细的语义)**当前 base.py 里都不存在了** — 项目自己把它们删掉了
- 11 个子类里**只有 1 个真正使用 conds 链**(`Empty_Ind`,纯粹当 AND 门用)

**结论**:所谓"链式合成 DSL"在生产里是个退化的扁平 AND 门。**真正干活的形态识别**(像"放量 → 回踩 → 企稳 → 进场"这种 4 状态切换的复杂形态)**全都是子类里手写的 Python 状态机**,不走 conds 字典。

### 发现 2:第一轮研究推荐的 Stage 2 建立在错觉之上

第一轮推荐的 "Stage 2 = 移植 ChainCondition + post_event_lookforward 因子",是把"链式合成"当成 Condition_Ind 的核心价值。但生产实证表明这层抽象**项目自己都不深用**,移植它就是在抄一个项目自己已经放弃的弯路。

**Stage 2 应该被砍掉重写**。

### 发现 3:两个项目其实在做不同的事

| | BO 框架(Trade_Strategy) | Condition_Ind(new_trade) |
|---|---|---|
| 计算模型 | 离线批处理 | 流式逐 bar |
| 输出 | 事件 + 标量(进挖矿) | 时序信号(进 backtrader 下单) |
| 解决的问题 | 离线挖掘 + 评分规则生成 | 在线触发 + 回测进出场 |

**真正的功能重叠只有 2 项**:放量判定 + 形态可视化。其他要么互补,要么"长得像但数学上不等价"。

## 六、新方案:三层工具集(替代 ChainCondition)

修订后的 Stage 2 不是移植 Condition_Ind,而是在 pandas 上**自己搭一套表达力相当但更轻量的工具集**。

**厨房比喻**:

| 工具 | 比喻 | 适合做什么 |
|---|---|---|
| **底层:DataFrame 谓词代数** | **菜刀** — 最常用、最快、95% 的菜都靠它 | 写"过去 N 天有没出现 X" / "X 与 Y 同时成立" / "先 X 再 Y" 这类 vectorized 判定。直接用 pandas 的 `&` `|` `rolling` `shift`,无 AST、无解释器,代码量是 Condition_Ind 的 1/3。 |
| **中层:BO-anchored 窗口原语** | **量杯** — 专门量"事件触发那一刻的某个值",再后续比较 | 写"BO 之后 10 天 close 不破该 BO 当根 high 的 95%"这种**动态依赖**(95% 是逐 BO 不同的常量)。用 pandas `groupby + transform('first')` 把"事件触发 bar 的某常量"广播到事件之后的窗口。 |
| **顶层:Python 状态机类** | **慢炖锅** — 不快但能煮复杂菜 | 写"放量 → 回踩 → 企稳 → 触发"这种 4 状态切换、可中途回退、要记忆动态变量的复杂形态。直接 Python `for idx in df.index` 写状态机,不挂 backtrader,产物包装成 BO 行级标量进挖矿。 |

**三个工具的产物都是 `pd.Series`**:
- 进挖矿 → `series.iloc[bo_idx]` 取标量
- 进实盘 → `series.iloc[-1]` 取最新值
- **共享同一计算层,但取样时刻不同**

### 用户的"Platform-as-Event"想法在这套工具集下怎么落地

形态 A(推荐,简单情况):

```python
@feature(name='platform_formation', lookback=K+W)
def platform_formation(df, K=10, eps=0.03, N=15):
    bo_in_past = df['is_bo'].rolling(N).max().astype(bool)
    price_stable = (df.close.rolling(K).std() / df.close.rolling(K).mean()) < eps
    above_resistance = df.close > bo_anchored(df['is_bo'], df['high'])
    return bo_in_past.shift(K) & price_stable & above_resistance

# 然后写一个轻量 PlatformDetector,扫描 series 上升沿生成 platform 事件 row
# 现有挖矿流水线零修改 — 仅替换"事件输入源"
```

形态 B(语义复杂时):写一个 Python 状态机类,逻辑同 new_trade 的 `BreakoutPullbackEntry`,只是不挂 backtrader。

## 七、最新对话:`exp` 的真正含义

最后一轮对话,用户指出团队对 Condition_Ind 的 `exp` 字段理解错了:

- 团队解读为"**窗口聚合**:过去 N 根任一根满足"
- 用户本意是"**过期时间(expiration)**:事件 A 发生后,作为其他事件前提条件的有效期是 `exp` 根 bar"

**例子**:"放量之后,必须在 10 天内上涨,否则放量不可作为上涨的条件(距离太远,关联弱)"

机械上两者等价(都是 `s.rolling(exp).max().astype(bool)`),但**概念上**:

| Framing | 提出的问题 |
|---|---|
| 窗口聚合(团队解读) | "过去 N 根**有没有发生过** A?" |
| **过期时间**(本意) | "A **还在不在它的有效期内**,可以作为 B 的前提条件吗?" |

### 这个修正影响什么

修正后,底层 API 应该把两种语义**显式分开**,而不是塞进一个统一的 `mode` 参数:

```python
# 事件过期类 — 询问"事件 A 是否还有效"
expires_within(event_mask, exp=10)

# 状态持续类 — 询问"状态 X 是否稳定"
state_persistent(state_mask, n=40, mode='all'|'ratio'|'consecutive')
```

这两类原语在挖矿里都是"BO 行 iloc 取标量",对外接口一致;但**内部语义清晰二分**。

### `exp_cond` 的重新理解

旧 `Result_ind`(已删除)有一个 `exp_cond` 字段,语义是"在 exp 有效期内还要叠加另一个条件"。被删可能是因为**塞在 cond 字典里语义混乱**,不是因为语义不需要。

在三层工具集下,这种语义可以用 BO-anchored + 谓词代数的简单 `&` 组合优雅表达:

```python
# vol_burst 在过去 10 天内发生 + 那次发生时 RSI 在 [40, 60]
qualifying_bursts = vol_burst & (rsi >= 40) & (rsi <= 60)
recent_qualifying = expires_within(qualifying_bursts, exp=10)
```

## 八、一句话总结

> **现阶段:加一个 `ma_flat` 因子,跑挖矿,看效果。**
> 别移植 Condition_Ind,别上 MATCH_RECOGNIZE,别引入 ChainCondition 抽象层。
> 等 OOS 验证或新规律出现的时候,再按需引入"三层工具集"中合适的那一层。
> Platform-as-Event 是好想法,等真要做的时候,用三层工具集的形态 A 自然落地。
