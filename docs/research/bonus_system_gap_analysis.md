# Bonus 系统缺口分析报告

> 基于 9810 个美股突破事件的统计分析结果，审视当前评分系统的关键缺口，评估新增 bonus 的必要性及无效因子的处置方案。

---

## 一、当前 Bonus 系统现状

当前评分器采用乘法模型，共 12 个因子（`height_bonus` 已禁用）：

## 二、关键缺口分析

### 缺口 1：Streak 方向完全错误（P0）

**现状**：Streak 作为 bonus 加分（×1.20 / ×1.40），但数据显示它是**所有因子中相关性绝对值最大的负相关因子**（r = -0.1988, p = 5.03e-88）。

**数据证据**：
- level 0 median = 0.1956 → level 2 median = 0.1120，频繁突破的股票后续收益**下降 43%**
- RF 重要性排名第二（0.2178），区分力极强
- PK-Mom + Streak 是唯一的负交互对（effect = -0.0327）

**可能原因**：
- 均值回归效应：频繁突破 = 多轮上涨后接近阶段性顶部
- 阻力位质量稀释：短期频繁突破说明阻力位密度高、间距小，单次突破意义被稀释
- 筹码结构恶化：连续突破吸引大量短线跟风盘，获利盘堆积导致后续抛压加重

**建议**：将 Streak 从 bonus 改为 **penalty**，乘数反转为 < 1.0（如 ×0.85 / ×0.70）。这是本次分析中**预期影响最大的单一改动**。

### 缺口 2：Drought 需要非线性处理（P1）

**现状**：Drought 线性递增（×1.15 → ×1.30），但数据显示非线性特征。

**数据证据**：
- level 0 median = 0.1573
- level 1 median = **0.2489**（+58.2%，最佳）
- level 2 median = 0.2161（回落）
- level 3 median = 0.1907（继续回落）

**解释**：适度的"沉寂期"（约 60~80 交易日）后的突破效果最佳，过长的沉寂反而信号衰减——可能意味着股票本身缺乏市场关注度。

**建议**：改为倒 U 型乘数设计：level 1 保持 ×1.20~1.30（最高），level 2/3 回退到 ×1.10~1.15 甚至 ×1.0。

### 缺口 3：Overshoot 角色矛盾（P2）

**现状**：设计为惩罚（×0.80 / ×0.60），但统计显示 level 1/2 的 median（0.2771 / 0.2923）**远高于** level 0（0.1593）。

**解释**：幸存者偏差——能在突破后 5 天内产生足够大 gain\_5d 以触发 Overshoot 的股票，本身就是动量最强的那批。Overshoot 触发实质上是在筛选"极强动量突破"，虽然短期有回调风险，但 10~40 天窗口内仍保持了较好的累积收益。

**建议**：暂时将 Overshoot **设为中性**（乘数 = 1.0）或 `enabled: false`，避免错误地惩罚高动量突破。后续需设计更精细的超涨判定（如分离 gap 超涨和日内超涨）。

## 三、是否需要新增 Bonus？

### 3.1 候选：突破 K 线形态因子（breakout\_type）

**现状**：`Breakout` 对象已计算 `breakout_type`（yang / yin / shadow），但从未纳入评分。

**分析**：从技术分析直觉看，阳线突破（yang）比阴线突破（yin）更可靠，但当前统计报告中未包含该维度的数据。

**建议**：**暂不新增**，优先在 `bonus_combination_analysis.py` 的 `build_dataframe()` 中加入 `breakout_type` 列，先跑一轮统计验证。只有数据支持时才考虑新增。

### 3.2 候选：交互 Bonus（Volume × PBM 等）

**现状**：交互效应分析显示 Volume 与多个因子存在强正交互：
- Volume + PBM: effect = 0.2114
- Volume + Drought: effect = 0.2097
- Volume + Streak: effect = 0.1950

**分析**：在**乘法模型**中，交互效应已被自然捕获——当 Volume ×1.30 和 PBM ×1.30 同时触发时，总分已自动变为 ×1.69，比单独触发更高。这正是乘法模型相比加法模型的结构性优势。

**建议**：**不需要新增交互 Bonus。** 确保核心因子的单因子权重合理即可，乘法模型会自动处理组合增益。

### 3.3 候选：其他未使用的已有特征

对 `Breakout` dataclass 和 `FeatureCalculator` 的审查结果：

| 已有特征 | 是否纳入评分 | 新增价值评估 |
|---------|------------|------------|
| `breakout_type` | 否 | 待验证（见 3.1） |
| `stability_score` | 否 | **不可用** — 使用了未来数据（前瞻偏差） |
| `daily_return_atr_ratio` | 间接（DayStr 已覆盖） | 冗余，无需重复 |
| `gap_atr_ratio` | 间接（DayStr 已覆盖） | 冗余，无需重复 |
| `atr_normalized_height` | 可选（ATR-Height） | 已有机制，按需启用 |

**结论：当前代码中没有被遗漏的高价值特征。** 最大的改进空间在于修正现有因子，而非新增。

## 四、无效因子处置方案

### 4.1 为什么 Age、Tests、PeakVol 无效？

**Age（阻力位年龄）**：传统技术分析认为"越老的阻力位突破越有力"，但 9810 个样本不支持这一观点。
- 各 level 的 median 几乎无差异（0.1514 ~ 0.1783）
- 可能原因：远古阻力位的当时持仓者大部分已离场，所谓"套牢盘解放"的叙事在实际中不成立

**Tests（测试次数）**：传统认为"被测试越多次越可靠"，但数据完全不支持。
- p = 0.70，完全不显著
- 可能原因：test\_count 基于 3% 价差阈值的贪心聚类，定义噪音大；测试次数多也可能意味着犹豫不决

**PeakVol（峰值放量）**：效果不稳定。
- level 1（peak 放量 2~5x）的 median 反而低于 level 0
- 仅 level 2（peak 放量 > 5x）略有提升，但样本量有限（n=362）

### 4.2 能否直接移除？

**关键发现：Age 和 Tests 不能简单移除。**

`_classify_pattern()` 中的模式判定直接依赖 Age 和 Tests 的 level 值：

```python
# Age 参与的模式判定
if pk_mom_level >= 1 and age_level >= 2:
    return "deep_rebound"       # 深蹲远射
if vol_level >= 1 and age_level >= 2:
    return "power_historical"   # 放量历史突破（最佳模式！median=0.2981）
if age_level >= 2:
    return "historical"         # 历史阻力突破

# Tests 参与的模式判定
if age_level >= 2 and tests_level >= 1:
    return "grind_through"      # 磨穿防线
if tests_level >= 2:
    return "dense_test"         # 密集测试
```

如果移除 Age，`power_historical`（统计分析中表现最佳的模式，median = 0.2981）将**无法被识别**。

### 4.3 各因子处置建议

| 因子 | 建议 | 具体操作 | 理由 |
|------|------|---------|------|
| **Age** | **降权保留** | 乘数降至 ×1.02 / ×1.05 / ×1.08 | 评分影响趋近于零，但保留 level 计算以支撑模式标签 |
| **Tests** | **降权保留** | 乘数降至 ×1.02 / ×1.05 / ×1.08 | 同上，支撑 `grind_through` 和 `dense_test` 模式 |
| **PeakVol** | **禁用** | 设 `enabled: false` | 不参与 `_classify_pattern` 的任何判定逻辑，可安全禁用 |

### 4.4 代码影响范围

通过全代码库搜索，`age_bonus`、`test_bonus`、`peak_volume_bonus` 仅在以下位置被引用：

| 文件 | 用途 | 改动影响 |
|------|------|---------|
| `breakout_scorer.py` | 核心评分逻辑 | 调整乘数值 / enabled 标志 |
| `UI/config/param_editor_schema.py` | UI 参数编辑器 schema | 无需改动（降权只改值） |
| `UI/config/param_loader.py` | 参数加载 | 无需改动 |
| `configs/params/scan_params.yaml` | 配置文件 | 修改阈值/乘数 |

没有被 observation pool、daily pool 或其他分析模块直接依赖。**调整影响范围完全可控。**

## 五、行动优先级总结

| 优先级 | 行动 | 类型 | 预期影响 | 实施难度 |
|--------|------|------|---------|---------|
| **P0** | Streak bonus → penalty（乘数反转为 ×0.85 / ×0.70） | 逻辑修正 | **最高** | 低 |
| **P1** | Drought 改为非线性（level 1 最高，level 2/3 回落） | 优化 | 中 | 低 |
| **P1** | Age / Tests 乘数降至 ×1.02~1.05（保留但几乎无影响） | 降权 | 低~中 | 低 |
| **P1** | PeakVol 设为 `enabled: false` | 禁用 | 低 | 低 |
| **P2** | Overshoot penalty 暂时禁用或设为中性 | 重评估 | 低~中 | 低 |
| **P3** | 在统计管道中加入 `breakout_type` 列，收集数据 | 数据收集 | 待验证 | 低 |

---

## 六、核心结论

1. **当前系统不需要新增 bonus 因子**——最大的改进空间在于修正现有因子的方向和权重
2. **Streak 逻辑反转是最高优先级改动**——将已确认的最强负信号从加分改为扣分
3. **Age 和 Tests 不能直接移除**——它们支撑模式标签系统，正确做法是降权到近乎中性
4. **PeakVol 可以安全禁用**——不参与任何下游逻辑
5. **乘法模型的结构优势意味着不需要交互 bonus**——组合增益已被自动处理
