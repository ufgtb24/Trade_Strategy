# 文档管理说明

本目录包含突破策略系统的所有技术文档，采用**三层文档体系**管理项目开发。

---

## 文档结构

```
docs/
├── plans/              # 规划文档
│   └── initial_plan.md
├── module_plans/       # 设计文档（01-10编号）
│   ├── 01_数据层设计.md
│   ├── 02_技术分析模块设计.md
│   ├── 03_搜索系统设计.md
│   └── ...（共10个模块）
└── module_summaries/          # 总结文档（与设计文档一一对应）
    ├── 02_技术分析模块总结.md  ✅ 已完成
    └── ...（开发完成后创建）
```

**对应关系示例**：
```
module_plans/02_技术分析模块设计.md  ←→  module_summaries/02_技术分析模块总结.md
    （开发前编写）                          （开发后创建）
```

---

## 三类文档的作用

### 1. 规划文档（plans/）

**用途**：项目宏观规划，定义模块划分和开发阶段

**核心文件**：
- `initial_plan.md` - 系统总体规划
  - 模块总览和依赖关系
  - 开发阶段划分（无时间预设）
  - 标注各模块完成状态

**更新时机**：
- 模块开发完成后标注状态
- 项目架构调整时

**特点**：
- 着重宏观逻辑，不包含实现细节
- 通过 `@` 引用指向详细设计文档

---

### 2. 设计文档（module_plans/）

**用途**：模块详细设计，开发前编写

**命名规范**：`XX_模块名设计.md`（XX为两位数字编号，01-10）

**必须包含**：
1. 模块概述（职责、目标、依赖）
2. 架构设计（子模块划分）
3. 接口定义（输入输出格式）
4. 核心算法（流程、伪代码）
5. 数据结构定义
6. 异常处理策略
7. 性能考虑
8. 测试方案

**更新时机**：
- 开发前编写
- 设计调整时更新

**作用**：
- 指导代码实现
- 作为开发蓝图和评审依据

---

### 3. 总结文档（module_summaries/）

**用途**：模块开发完成后的经验总结

**命名规范**：`XX_模块名总结.md`（**与设计文档编号一一对应**）

**命名示例**：
- 设计文档：`02_技术分析模块设计.md`
- 总结文档：`02_技术分析模块总结.md`

**必须包含**：
1. **核心成果** - 实现了什么功能
2. **与设计的差异** - 实现过程中的调整（如果有）
3. **测试结果** - 验证数据和性能指标
4. **经验教训** - 遇到的问题和解决方案
5. **使用示例** - 代码示例和最佳实践
6. **调参建议** - 关键参数说明

**更新时机**：
- 模块开发完成后立即创建
- 发现问题或优化后补充

**作用**：
- 记录实现细节和设计决策
- 为后续模块提供参考
- 与设计文档互补（设计=计划，总结=实际）

**重要原则**：
- ⚠️ **每个设计文档必须对应一个总结文档**
- ⚠️ **编号必须保持一致**（方便查找和追踪进度）
- ⚠️ **总结文档的存在表示该模块已完成开发**

## 文档管理策略

### 开发工作流

```
1. 开发前
   └─ 编写 module_plans/XX_模块名设计.md
        ↓
2. 开发中
   └─ 按照设计文档实现代码
        ↓
3. 开发后
   ├─ 创建 module_summaries/XX_模块名总结.md（编号与设计文档一致）
   └─ 更新 plans/initial_plan.md 标注完成状态
```

### 模块完成后的更新示例

**1. 创建总结文档**（**编号必须与设计文档一致**）

```
module_plans/02_技术分析模块设计.md  →  module_summaries/02_技术分析模块总结.md
```

参考现有 `module_summaries/02_技术分析模块总结.md` 的格式：
- 核心思想和算法演进
- 详细算法流程
- 测试结果对比
- 使用示例
- 关键参数说明

**2. 更新 initial_plan.md**

在"三、核心模块概览"中标注状态：

```markdown
### 3.2 技术分析模块（analysis/）
实现凸点识别、突破检测、质量评分等核心算法。**✅ 已完成（v2.0）**
→ 详见 @BreakthroughStrategy/docs/module_plans/02_技术分析模块设计.md
→ 总结 @BreakthroughStrategy/docs/module_summaries/02_技术分析模块总结.md
```

## 文档维护原则

1. **单一信息源**
   - 每类信息只在一个地方维护
   - 规划看 plans/，设计看 modules/，总结看 module_summaries/

2. **引用而非复制**
   - 使用 `@` 引用指向详细文档
   - 避免重复内容导致不一致

3. **及时更新**
   - 模块完成后立即更新状态
   - 设计调整时同步更新文档

4. **保持简洁**
   - 规划文档只写宏观逻辑
   - 实现细节放在设计和总结文档中
