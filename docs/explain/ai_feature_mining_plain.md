# 让 AI 帮我们总结 K 线形态 — 通俗说明

> 这是 `docs/research/feature_mining_via_ai_design.md` 的人话版。原文档约 700 行、含完整 schema 和伪代码；这里只讲**想法**和**为什么这么做**，不关心字段细节。

---

## 1. 我们要解决什么问题

现在整个选股流程是这样：

> 检测突破点 → 观察 K 线 → **人工总结形态特征** → 写成因子 → mining 挖阈值 → live 盯盘

**瓶颈卡在"人工总结形态特征"这一步**。你看 K 线时，能一眼看出"这是底部盘整后的温和突破"（好）或"这是已经涨了一大段的高位冲顶"（差），但很难用语言把这种直觉讲清楚，更难翻译成因子代码。

结果就是：live 里现在匹配出来的突破，很多是后者（图 B 那种高位冲顶），不是你想要的前者（图 A 那种底部盘整）。

**想法：把"总结形态"这活儿交给 AI**。注意这里的"AI"特指 Claude Code 本身，不是深度学习（上次用过会过拟合）。

---

## 2. 核心思路：让 AI 看一堆"好样本"，找共同点

你作为用户：
- 在 dev UI 里手动挑一批"好的突破"作为**正例**（按 `P` 键收集）
- 再挑一批"看着不好的突破"作为**反例**（按 `N` 键收集）

然后 skill 自动把这堆样本塞给 Claude，让它回答：
> "这批好样本有什么共同点？又怎么区别于坏样本？"

AI 吐出一份形态特征描述（文字），再自动翻译成一份**因子草稿**（yaml），你 review 后喂给现成的 `add-new-factor` skill 落地，最后交给 `mining/` 流水线做统计验证。

---

## 3. 怎么把 K 线喂给 AI？

这是本方案最关键的设计决定：**图 + 文 混合，各司其职**。

### 直觉：人看 K 线时是怎么工作的？

- **眼睛**识别形态：哦这是个矩形盘整、这是个 U 底、这是冲顶回落
- **大脑**抽取数字：盘整了多少天、振幅多大、突破放量几倍

### AI 也一样，分两个通道：

**图像通道（负责"看形状"）：**
- 给它一张干净的 K 线图
- 图上**只保留形态相关元素**：K 线、成交量、盘整区的黄色框、被突破的阻力线、BO 标记
- **删掉**所有会干扰的东西：MA 均线、网格、日期、股票代码、小字数值
- AI 看完会说："这是典型的矩形盘整突破"——定性

**文本通道（负责"报数字"）：**
- 一行鸟瞰摘要（含盘整核心指标 + `traj=` 自然语言轨迹标签，例如 `traj=bottom_lift→breakout`）
- 盘整段的 13 个关键数字（持续天数、振幅、上下沿各触碰几次、量能收缩比 ……）
- 现有 13 个因子的原值和档位
- AI 看完会说："盘整 55 天、振幅 8.8%、突破日量能 3.6 倍"——定量

> 注：早期方案中有一段 Zig-Zag 符号串（像 "D2L U1S D2M …"）。Review 后被**移除**——这种离散 token 序列更适合专门训练过此类 token 的小模型；对 Claude 这种通用 LLM，每看一个符号都要查 legend，推理成本高且易错。走势骨架改由**图像**（混合模式）或 Seg 1 的 `traj=` **自然语言短标签**（text-only 模式）承担。

### 为什么不偷懒用纯图像或纯文字？

- **纯图像不行**：AI 从图上读数字**不靠谱**（实测误差 ±30%）。图像只能看形状。
- **纯文字不行**：数字堆里 AI 很难抽象出"这是矩形盘整"这种人脑一眼看出的形态。
- **混合最佳**：图像给形态命名，文字提供可量化的阈值。两者的输出拼起来才是可执行的因子规则。

**强制契约：AI 永远不许从图像读数字。** 数字一律走文本通道。

---

## 4. 一次归纳不够，要多轮迭代

如果一次性塞 50 个样本让 AI 总结，它会记不全、抓不准。**所以分多轮，每轮 12 个样本左右**。

### 怎么避免"看了后面忘了前面"？

引入一个叫 `PartialSpec` 的**中间结论本**（yaml 格式）：

```
Round 1 输入：12 个正例 + 12 个反例
Round 1 输出：PartialSpec v1 — 初步不变量清单
           例如 "invariant 1: 盘整持续 30-80 根"
                "invariant 2: 放量比 ≥ 2.0"

Round 2 输入：PartialSpec v1 + 新的 12 个样本
Round 2 输出：PartialSpec v2 — 有的 invariant 被证伪，有的被 refine
           例如 "invariant 1 改为 40-80 根，发现 30 根太松"

...

直到连续两轮没有任何 invariant 变化 → 收敛，停止。
```

**关键设计**：中间结论本里记录了"哪些样本支持哪条规律"、"哪些样本反例证伪了哪条规律"。下一轮 AI 看到后不能"悄悄改结论"，必须留下修改痕迹。这防止 AI 滑向空想。

### 两段式 label 揭示（防作弊）

AI 很容易偷懒：看到 label 大就把它当成"正例"的特征反向拟合。

所以每轮分两步：

- **第一步盲猜**：给 AI 样本但**不给** label，让它纯靠形态归纳
- **第二步揭示**：给出真 label，让它自我纠偏（"我本来猜样本 S8 是好形态，但真 label 偏低，说明我的规律遗漏了什么"）

这样 AI 就**被迫先用眼睛看形态**，再用数字校准。

---

## 5. 反例怎么选？有个反直觉的细节

直觉上，反例应该挑"没涨的突破"对吧？但这样做会出问题：

- "没涨"的原因五花八门（行业背景差、大盘下跌、业绩暴雷……）
- AI 会把这些**无关噪声**当成"反例特征"，得出奇怪的结论
- 这种分析其实是 `mining/` 的 Spearman 方向诊断已经在做的事，重复了

**正确做法**：反例 = **label 很高但形态不是你想要的那种**。

例子：图 B 那种"高位冲顶快速突破"，label 可能也很大（短期内确实涨了），但你知道它形态差。拿它当反例，AI 就必须在**形态层面**给出判别——"正例有明显盘整区，反例没有"——这才是你真正想要的因子。

## 6. AI 产出什么？

两层输出，都写到 `docs/research/feature_library/<form_name>.md`：

### Layer A：自然语言描述（给人看）

```markdown
# Tight Rectangle Basing Breakout

盘整 40~80 根 K 线，振幅 5~15%，上下沿各至少触碰 2 次，
盘整期间量能收缩（RV < 1.0），突破日放量 ≥ 2x。
负例通常缺少"量能先收缩后放大"的节奏，且盘整斜率偏上（漂着往上走）。
```

读起来像交易员写的形态备忘录，人类能直接理解。

### Layer B：因子草稿（给 `add-new-factor` 吃）

```yaml
factor_draft:
  key: "range_tight"
  name: "Range Tightness"
  cn_name: "盘整紧致度"
  category: "context"
  default_thresholds: [0.015, 0.025, 0.04]
  default_values: [1.25, 1.15, 1.05]
  mining_mode: "lte"
  ...
```

这个 yaml 结构**正好**是 `add-new-factor` skill 的输入格式。你 review 过后直接跑那个 skill，就能在 3 个文件里完成因子注册。

---

## 7. 谁来判断新因子靠不靠谱？

**不是 AI，是 mining 流水线。**

AI 只负责"产生假设"（写出 Layer B）。真正验证这个假设靠不靠谱，走的是现成的：

```
Spearman 方向诊断 → TPE 阈值优化 → 5 维 OOS 验证
```

通过 → 因子留下；不通过 → 删掉就是（只改 3 个文件，成本极小）。

**这就是为什么这套方法不怕过拟合**：AI 写的东西再离谱，也过不了 OOS 验证这一关。mining 流水线是 falsifier（证伪器），AI 是 hypothesizer（假设生成器），两者分工。

---

## 8. 把这套流程包装成 skill

所有这些（挑样本 → 分批喂 → 多轮 compact → 产出两层输出）封装成一个 Claude Code skill：

```
induce-formation-feature/
  SKILL.md              # 主入口
  schemas/              # PartialSpec + FactorDraft 的 JSON Schema
  prompts/              # 3 个 prompt 模板（归纳、揭示、SYSTEM 条款）
  templates/            # 两层输出模板
```

**注意**：skill 里**不调用任何 API**。它只是一份"工作流程脚本"，由 Claude-the-agent（也就是你在 Claude Code 里会话的那个 AI）按照脚本执行。和现有的 `add-new-factor` 架构完全一样。

---

## 9. 落地顺序（推荐分 4 个 phase）

**Phase 1 — 先让结构化路径跑通**（最重要，最早可验证）
1. 给盘整段加 5 个字段（持续 bar 数、下沿触碰次数、斜率、紧致度、量能收缩比）
2. Corpus exporter（一个 BO → 一份三段式文本，含 Seg 1 的 `traj=` 自然语言轨迹标签）

做完这 2 件，就可以**不靠 UI、不靠 skill，手动喂 AI 跑第一次归纳**，看效果值不值得继续投。

**Phase 2 — UI 便利 + skill 骨架**
3. Dev UI 里加 P/N 键盘 hook
4. Cold-start label 分位抽样器
5. 写 skill 本体

这时候能端到端跑 `text_only` 模式。

**Phase 3 — 加图像**
6. AI-friendly 渲染函数 `render_bo_for_ai(bo, df, mode)`

之后可以启用 `hybrid_light` 主力模式。

**Phase 4 — 长期增强**（可选）
- Feature library 自动索引
- 多轮 PartialSpec 持久化缓存
- Sparse 聚类入口

---

## 10. 一句话总结

> **让你挑好/坏样本 → 自动渲染成干净的 K 线图 + 结构化数字 → Claude 多轮总结形态共性 → 产出一份可直接落地的因子草稿 → 由 mining 流水线做经验证伪。**

AI 做它擅长的事（识别形态、语言化归纳），mining 做它擅长的事（统计验证），分工清晰，互不踩脚。
