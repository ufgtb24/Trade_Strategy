# 日志化调试工作流

## 核心理念

将调试从「手动交互 + 观察」转变为「可重复执行 + 日志输出」。

```
常规流程：入口（UI/API/命令行）→ 用户交互 → 触发功能 → 输出（界面/响应/文件）
调试流程：配置文件 → 可重复的调试入口 → 直接调用功能 → 日志输出
```

调试入口（脚本、命令、测试）绕过或简化交互层，直接调用被测功能，通过日志完整表征程序状态。

## 截图使用原则

**可以接受截图作为调试的起点**：用户提供的截图有助于理解问题现象、确定调试方向。

**调试过程中不需要反复截图验证**：一旦开始调试，通过脚本和日志输出来跟踪状态，不要请求用户反复截图确认中间结果。

```
✓ 正确：用户提供截图描述问题 → 日志调试 → 日志验证 → 完成
✗ 错误：截图 → 修改 → 再截图 → 修改 → 再截图 → ...
```

## 工作流程

### 第一步：分析问题范围

先定位问题涉及的功能模块：

1. **追溯调用链**：从入口点追溯到底层功能，确定问题可能出现的层级
2. **缩小范围**：确定最小可复现问题的功能模块
3. **明确边界**：确定需要调用的入口函数和需要观察的输出

常见调用链模式：
- UI 项目：界面事件 → 事件处理 → 业务逻辑 → 数据层
- 后端服务：API 入口 → 控制器 → 服务层 → 数据访问
- 数据处理：输入解析 → 转换处理 → 计算逻辑 → 输出格式化

### 第二步：判断调试方式

根据功能的调用方式，选择合适的调试入口：

**可以直接调用的情况（不需要创建脚本）**：
- 命令行工具：直接运行命令即可
- API 接口：通过 curl、httpie 或现有客户端调用
- 已有测试用例：运行现有的单元测试或集成测试
- 可独立运行的模块：`python -m module` 或 `python module.py`

**需要创建调试脚本的情况**：
- 需要绕过 UI/交互层才能触发的功能
- 需要模拟特定的参数组合
- 需要隔离测试某个内部函数
- 现有入口不方便重复执行或参数不易控制

```
判断流程：
1. 功能是否可以通过命令行/API/现有测试直接触发？
   → 是：直接使用，跳到第四步
   → 否：继续判断
2. 是否需要绕过交互层或模拟特定参数？
   → 是：创建调试脚本（第三步）
   → 否：直接在代码中添加调试输出（第四步）
```

### 第三步：创建或复用调试脚本（如需要）

仅当第二步判断需要脚本时执行此步骤。

先检查 `scripts/debug/` 目录是否有可复用的脚本：

```bash
ls -la scripts/debug/
grep -r "关键词" scripts/debug/
```

如果存在可复用的脚本，优先修改或扩展，不要重复创建。

创建新脚本时：
- 存放位置：`scripts/debug/`
- 命名规范：`debug_<功能模块>_<具体功能>.py`
- 脚本要求：
  - 加载项目配置文件，保持与用户环境一致
  - 直接调用第一步定位的功能模块入口
  - 模拟交互层传入的参数
  - 输出足够的中间状态信息

### 第四步：在功能代码中添加调试输出

在被调试的功能代码中添加临时日志，完整表征问题：

```python
# 在关键位置添加调试输出
print(f"[DEBUG] 函数入口 - 参数: {args}")
print(f"[DEBUG] 中间状态 - 变量: {var}")
print(f"[DEBUG] 条件分支 - 进入: {'分支A' if cond else '分支B'}")
print(f"[DEBUG] 函数出口 - 返回: {result}")
```

调试输出原则：
- 标记 `[DEBUG]` 前缀，便于识别和搜索
- 覆盖函数入口、出口、关键分支、状态变化点
- 输出变量的实际值，不只是类型
- 对于复杂对象，输出关键属性而非整个对象

### 第五步：自动化调试循环

执行「修改 → 运行 → 检查」循环，直到输出符合预期：

```
循环流程：
1. 运行调试入口（脚本/命令/测试）
2. 分析日志输出
3. 如果输出不符合预期：
   a. 定位问题代码
   b. 修复代码
   c. 返回步骤 1
4. 如果输出符合预期：
   a. 结束循环
```

## 重要规则

### 必须遵守

1. **不要反复请求截图** - 截图可作为起点，但调试过程依赖日志而非截图
2. **先判断是否需要脚本** - 能直接运行的功能不需要额外创建脚本
3. **不要重复创建脚本** - 需要脚本时，先检查 `scripts/debug/` 是否有可复用的
4. **使用项目配置文件** - 保持与用户实际使用环境一致
5. **调试入口必须可重复执行** - 无论是脚本、命令还是测试，都要能反复运行

### 配置文件优先

调试时优先使用项目的配置文件路径，常见位置：
- `config.yaml` / `config.json`
- `settings.py`
- `.env`
- `configs/` 目录

原因：配置文件是用户与开发者沟通的共同基础，使用相同配置可以复现用户遇到的问题。

### 何时需要截图

**适合用截图的场景**（作为调试起点）：
- 用户描述问题现象时
- UI 布局/样式/渲染异常
- 需要理解用户看到的具体状态

**不需要截图的场景**（用日志替代）：
- 调试过程中的中间验证
- 确认功能逻辑是否正确
- 检查数据流和状态变化

原则：截图帮助理解问题，日志用于解决问题。

```
└── ...
```

## 示例场景

### 场景一：UI 功能问题（需要脚本）

**问题**：用户截图反馈「点击按钮后数据没有保存」

**判断**：保存功能由 UI 按钮触发，无法直接通过命令行调用 → 需要创建脚本

**正确做法**：
1. 接受截图或问题，理解问题现象
2. 分析问题范围：UI 按钮 → 事件处理 → 保存函数 → 数据层，定位到保存函数为调试入口
3. 检查 `scripts/debug/` 是否有保存功能的调试脚本，没有则创建
4. 在保存函数中添加 `[DEBUG]` 日志
5. 运行脚本，分析日志定位问题
6. 修复后再次运行脚本，通过日志确认修复成功

### 场景二：命令行工具问题（不需要脚本）

**问题**：用户反馈「运行 `python process.py --input data.csv` 输出结果不正确」

**判断**：功能本身就是命令行工具，可以直接运行 → 不需要创建脚本

**正确做法**：
1. 分析问题范围：参数解析 → 数据读取 → 处理逻辑 → 输出
2. 在相关函数中添加 `[DEBUG]` 日志
3. 直接运行 `python process.py --input data.csv`
4. 分析日志，定位并修复问题
5. 再次运行命令确认

### 场景三：API 接口问题（不需要脚本）

**问题**：API 返回错误数据

**判断**：可以通过 curl 或现有客户端直接调用 → 不需要创建脚本

**正确做法**：
1. 分析问题范围：路由 → 控制器 → 服务层 → 数据访问
2. 在相关层添加 `[DEBUG]` 日志
3. 用 `curl` 或测试客户端发送请求
4. 分析日志，定位并修复问题
