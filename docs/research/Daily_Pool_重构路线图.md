# Daily Pool 重构路线图

> 从"理论最优模板"到"最大公约数规则"的范式转换

---

## 一、问题诊断

### 1.1 当前系统的困境

**现象**：
- 能通过筛选的股票少得可怜（信号率 ~3%）
- 无法观察到规律，无法有效迭代
- 无法识别被过滤掉的"无辜好股票"

**根因分析**：

| 问题 | 表现 | 数学解释 |
|------|------|---------|
| 串联过滤 | 6个条件串联，通过率指数下降 | 若单条件通过率50%，6条串联后仅1.56% |
| 过拟合风险 | 28个参数 vs ~100样本 | 样本/参数比=3.6（应>10） |
| 路径依赖 | 必须按顺序经历各阶段 | 错过转换窗口即失败 |

### 1.2 方法论错误

**当前方法**：理论最优模板
- 预设"理想突破走势"应该是什么样
- 构造复杂公式拟合这个理想模板
- 不符合模板的都被过滤

**问题**：
- 权重来源不透明（0.4/0.3/0.3 从何而来？）
- 假设过强（所有成功突破都符合同一模板？）
- 无法从数据学习（规则是预设的，不是发现的）

## 二、新的设计哲学

### 2.1 最大公约数方法

> **从成功案例中提取共性特征，而非构造理论最优特征**

| 维度 | 理论最优 | 最大公约数 |
|------|---------|-----------|
| 规则来源 | 演绎（从理论到规则） | 归纳（从数据到规则） |
| 复杂度 | 高（精确量化每个因素） | 低（只保留高频特征） |
| 过拟合 | 高风险 | 低风险 |
| 可解释性 | 差（权重含义不明） | 好（每条规则对应市场机制） |

**学术支撑**：
- Occam's Razor：简单解释优于复杂解释
- Minimum Description Length：最优模型=最短编码
- Decision Tree Pruning：剪枝防止过拟合

### 2.2 由简入繁开发流程

> **从最简规则开始，通过分析失败案例逐步添加条件**

```
当前方法（由繁入简）:
  复杂规则 → 信号稀少 → 无法观察 → 无法迭代
       ↑___________________________|
              死循环

新方法（由简入繁）:
  简单规则 → 大量候选 → 分析失败案例 → 添加规则 → 精准度提升
                              ↑__________________|
                                   良性迭代
```

### 2.3 即时判断 vs 状态机

**核心洞察**：用户的规则是**瞬时条件**，不是**时序过程**

> "放量强势突破！稳住了！开始上涨？买入！"

这是在问"当前快照是否满足条件"，而不是"是否按顺序经历了各阶段"。

| 状态机 | 即时判断 |
|-------|---------|
| 必须经历 A→B→C→D | 任意时点满足条件即触发 |
| 错过转换窗口=失败 | 今天不满足，明天可能满足 |
| 记录历史状态 | 只看当前快照 |
| 阶段超时机制 | 单一观察期 |

## 三、MVP 版本设计

### 3.1 核心规则

```
触发买入信号的条件（同时满足）:

1. [前置] 入池质量达标
   quality_score >= min_quality_score

2. [稳住] 回调可控 + 未破支撑
   pullback_depth <= max_pullback_atr
   AND current_price > recent_support

3. [上涨] 放量阳线 + 价格上涨
   is_bullish_candle (close > open)
   AND volume >= volume_threshold * vol_ma20
   AND (close > yesterday_close OR close > ma5_close)
```

### 3.2 参数设计

**仅 4 个核心参数**（vs 当前 25+）：

| 参数 | 默认值 | 含义 |
|------|-------|------|
| `max_pullback_atr` | 1.5 | 最大回调深度（ATR单位） |
| `support_lookback` | 10 | 支撑参考天数 |
| `volume_threshold` | 1.3 | 放量阈值 |
| `min_quality_score` | 60 | 最小入池质量分 |

### 3.3 简化对照表

| 当前系统 | MVP 版本 | 简化方式 |
|---------|---------|---------|
| 7 状态状态机 | 无状态 | 改为即时判断 |
| 支撑位聚类+强度评分 | `min(low[-N:])` | 取近期最低价 |
| 收敛分数三因素公式 | 隐含在回调检查 | 合并指标 |
| 成交量趋势判断 | 仅保留放量检测 | 删除冗余 |
| 阶段转换条件矩阵 | 条件并行检查 | 去掉路径依赖 |
| 12字段证据结构 | 直接判断 | 去掉中间层 |

### 3.4 失败条件

```
移出观察池的条件（满足任一）:

1. 观察期满
   days_in_pool > max_observation_days (如30天)

2. 回调过深（给50%缓冲）
   pullback_depth > max_pullback_atr * 1.5
```


**产出**：
- 区分度最高的 3-5 个特征
- 每个特征的建议阈值

**执行者**：AI 分析 + 人类解读

### Phase 3: MVP 实现

**目标**：实现简化版评估器，与现有系统并行

**实现**：
```python
class SimpleDailyEvaluator:
    """MVP 版本：即时判断，无状态机"""

    def evaluate(self, entry, df) -> Optional[BuySignal]:
        if not self._is_stable(entry, df):
            return None
        if not self._is_rising(df):
            return None
        return self._generate_signal(entry, df)
```

**产出**：
- `SimpleDailyEvaluator` 类
- 配置开关选择使用哪个评估器

**执行者**：AI 开发

---

### Phase 4: A/B 对比验证

**目标**：用数据验证 MVP 版本效果

**指标**：
| 指标 | 说明 |
|------|------|
| 信号触发率 | 入池后成功触发的比例 |
| 信号质量 | 触发后 N 天收益率 |
| 假信号率 | 触发后跌破止损比例 |

**决策**：
- MVP 效果更好 → 替换现有系统
- 效果相当 → 保留 MVP（更简洁）
- 效果更差 → 分析原因，调整规则

**执行者**：人类运行回测 + AI 生成报告

---

### Phase 5: 迭代优化

**目标**：基于失败案例逐步添加规则

**流程**：
```
1. 收集 MVP 的失败案例（触发信号但后续亏损）
2. 分析失败案例的共性特征
3. 如果存在高区分度特征 → 添加为新规则
4. 验证新规则的 ROI（精准度提升 vs 召回损失）
5. 重复
```

**原则**：
- 每次只添加一条规则
- 每条规则必须有数据支撑
- 规则数量上限：5-7 条

**执行者**：人类决策 + AI 实现

| 风险 | 缓解措施 |
|------|---------|
| MVP 过于简化导致假信号增多 | 通过 Phase 5 迭代添加必要规则 |

## 八、附录：市场机制支撑

### MVP 规则的市场机制解释

```
[规则1] 入池质量达标
  市场机制: 放量突破 = 资金进场，供需失衡
  验证方式: 突破日成交量 >= 1.5x 基准量

[规则2] 稳住
  市场机制: 获利盘卖出 vs 新资金接盘 = 供需再平衡
  验证方式: 回调可控 + 未破支撑

[规则3] 上涨
  市场机制: 新资金确认突破有效，再次供需失衡
  验证方式: 放量阳线 + 价格上涨
```

### 为什么这比复杂公式更可靠

> "每一条规则必须对应一个独立的市场机制，且该机制的存在可被独立验证"
> — 金融版奥卡姆剃刀

复杂公式的问题：
- `strength = 0.4*(count/5) + 0.3*(span/15) + 0.3*bounce`
- 问：这对应什么市场机制？为什么是 0.4/0.3/0.3？
- 答：无法回答 → 可能是噪音拟合

简单规则的优势：
- `has_support = price > recent_low`
- 问：这对应什么市场机制？
- 答：价格未跌破近期低点 = 卖压有限，支撑有效
